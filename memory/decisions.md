# Журнал архитектурных решений (ADR)

Фиксируем ключевые решения проекта, чтобы не повторять ошибок и не забывать «почему так сделали».

---

## ADR-001: CORS-прокси вместо прямых запросов к WB API
- **Статус:** Принято
- **Дата:** 2026-04-10
- **Контекст:** Wildberries API блокирует все запросы из браузера (CORS). Без серверного посредника фронтенд не может получить данные.
- **Решение:** Создан `api/wb-proxy.ts` как Vercel Serverless Function. Все запросы из `wbService.ts` идут через него.
- **Последствия:** Добавляет ~100ms латентности. Зато работает из любого домена без настройки CORS.

## ADR-002: Supabase вместо localStorage
- **Статус:** Принято
- **Дата:** 2026-04-10
- **Контекст:** Данные (токены API, себестоимость товаров) терялись при очистке браузера или смене устройства.
- **Решение:** Миграция на Supabase. Таблицы `user_settings` (JSONB) и `product_cogs`. RLS включен.
- **Последствия:** Данные доступны с любого устройства после авторизации. Появилась зависимость от Supabase (бесплатный тариф: 500MB, 50K запросов/мес).

## ADR-003: Удаление framework: "vite" из vercel.json
- **Статус:** Принято
- **Дата:** 2026-04-10
- **Контекст:** Когда Vercel видит `framework: "vite"`, он берёт на себя сборку и ИГНОРИРУЕТ папку `api/` (не создаёт Serverless Functions). Все запросы к `/api/wb-proxy` возвращали 404.
- **Решение:** Убрали `framework`, явно указали `buildCommand` и `outputDirectory`.
- **Последствия:** Vercel теперь корректно обрабатывает и SPA (dist/), и Serverless Functions (api/).

## ADR-004: Отключение Confirm Email в Supabase
- **Статус:** Принято
- **Дата:** 2026-04-10
- **Контекст:** По умолчанию Supabase требует подтверждение email. Без настроенного SMTP-сервера письма не приходили, пользователи не могли зарегистрироваться.
- **Решение:** Подтверждение email отключено. Для создания тестового пользователя использовался service_role ключ через Admin API.
- **Последствия:** Упрощает регистрацию, но снижает безопасность. При необходимости можно включить обратно + настроить SMTP.

## ADR-005: Удаление демо-режима
- **Статус:** Принято
- **Дата:** 2026-04-10
- **Контекст:** Демо-режим с фейковыми данными (MOCK_SALES, MOCK_ORDERS) путал пользователя и создавал лишнюю ветку логики.
- **Решение:** Полное удаление демо-режима. При отсутствии токена дашборд показывает пустые графики.
- **Последствия:** Чистый UX. Но новый пользователь видит «пустоту» до ввода токена — нужен онбординг или подсказки.
